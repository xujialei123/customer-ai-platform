# AGENTS.md - Codex / AI 开发约束说明

本项目是一个「多平台 AI 客服中台」项目，用于统一接入企业微信客服、抖音来客、美团到店团购等客服入口，并通过 RAG 知识库 + OpenClaw/LLM 生成回复。

## 1. 项目目标

第一版 MVP 目标：

1. 企业微信客服走官方 Webhook/API，可以自动回复，但默认关闭自动发送。
2. 抖音来客、美团到店团购先使用 RPA Adapter 骨架，默认只生成建议回复，不自动发送。
3. 所有平台消息统一转换为 `UnifiedMessage`。
4. 所有 AI 回复必须基于知识库 RAG 检索结果。
5. 高风险问题必须转人工。
6. 代码需要结构清晰、注释充分、方便后续扩展。

## 2. 非常重要：Windows 中文乱码限制

为了避免 Windows 平台中文乱码，所有 AI Agent / Codex 在修改项目时必须遵守：

1. 所有源代码、Markdown、JSON、YAML、ENV 示例文件必须使用 **UTF-8 编码，无 BOM**。
2. 禁止使用 ANSI、GBK、GB2312 保存文件。
3. Node.js 读取文本文件时，必须显式指定编码：

```ts
const text = await fs.readFile(filePath, 'utf-8');
```

4. Node.js 写入文本文件时，也必须显式指定编码：

```ts
await fs.writeFile(filePath, content, 'utf-8');
```

5. 不要使用会依赖系统默认编码的写法。
6. Windows PowerShell 调试中文输出前，建议使用：

```powershell
chcp 65001
$OutputEncoding = [System.Text.Encoding]::UTF8
```

7. 如果需要生成日志文件，日志文件也必须使用 UTF-8。
8. 如果生成 CSV 文件，请优先使用 UTF-8 with BOM，仅 CSV 可以例外，因为 Excel 在 Windows 下识别 UTF-8 CSV 经常有问题。其他项目文件严禁 BOM。
9. Markdown 文件里的中文必须保持正常显示，不允许出现 `Ã¦`、`ä¸­æ`、`????` 这类乱码。
10. 如果 Codex 修改了文件，必须确保保存后的中文注释仍然可读。

## 3. 代码注释要求

1. 所有核心业务代码都必须写中文注释。
2. 注释要解释「为什么这样做」，不要只重复代码表面含义。
3. 平台适配器、RAG、风控、OpenClaw 调用、RPA 操作必须重点注释。
4. 不要删除已有中文注释，除非代码逻辑已经变化。
5. 新增复杂函数时，必须写函数级注释。

## 4. 架构原则

不要让 OpenClaw 直接对接各平台。必须通过本项目的 Adapter 层统一转换。

标准链路：

```text
平台 Webhook / RPA
  -> Platform Adapter
  -> UnifiedMessage
  -> MessageService 去重入库
  -> RAG 检索知识库
  -> OpenClaw / LLM 生成回复
  -> SafetyService 风控
  -> 自动发送或生成 ReplyDraft 人工审核
```

## 5. 平台策略

### 企业微信

- 走官方客服 API / Webhook。
- 当前项目提供占位实现。
- 默认 `AUTO_REPLY_ENABLED=false`，防止测试阶段误发消息。

### 抖音来客

- 第一版使用 RPA 骨架。
- 优先监听 WebSocket / XHR / DOM 文本。
- 不要优先使用截图视觉识别，因为 token 成本高且稳定性差。
- 默认只生成建议回复，不自动发送。

### 美团到店团购

- 第一版使用 RPA 骨架。
- 公开资料中外卖/闪购 IM 能力不能直接等同于到店团购客服。
- 默认只生成建议回复，不自动发送。

## 6. RAG 规则

AI 回复必须基于知识库检索结果。

知识库内容包括：

- 门店地址、电话、营业时间、停车信息。
- 套餐名称、价格、适用人数、使用时间、是否预约、不可用日期。
- 退款、改期、过期、核销失败等售后规则。
- 高频 FAQ。
- 禁止承诺和转人工规则。

如果知识库没有明确答案，必须回复：

```text
这个我帮您转人工确认一下。
```

## 7. 风控规则

以下内容必须转人工：

- 退款
- 投诉
- 差评
- 赔偿
- 食品安全
- 吃坏
- 过敏
- 报警
- 12315
- 工商
- 法律
- 律师
- 威胁、辱骂、情绪激烈

AI 不允许承诺：

- 退款
- 赔偿
- 免费赠送
- 特殊折扣
- 私下交易
- 绕开平台规则

## 8. Codex 开发顺序建议

请按 `tasks.md` 中的任务顺序开发，不要一次性生成大量不可运行代码。

优先保证：

1. 项目能启动。
2. 数据库能迁移。
3. API 能返回结果。
4. RAG 能检索。
5. ReplyWorker 能生成草稿。
6. 企业微信 Adapter 能接收测试消息。
7. RPA Adapter 后续逐步完善。

## 9. 禁止事项

1. 不要把平台私钥、token、cookie 写死在代码里。
2. 不要默认开启自动发送。
3. 不要让 AI 绕过知识库胡编答案。
4. 不要把 RPA 自动发送作为默认行为。
5. 不要在没有人工审核的情况下处理退款、投诉、赔偿类问题。
6. 不要删除本文件中的 Windows 编码限制。

## 10. 项目流程文档同步规则

1. `docs/project-flow.html` 是本项目面向开发、测试和交付人员的流程总览文档，必须保持可直接用浏览器打开。
2. 修改以下内容时，必须在同一个开发阶段同步更新 `docs/project-flow.html`：
   - 服务、端口、启动命令或 Docker 依赖。
   - 平台 Adapter、消息链路、WebSocket、RPA 多会话调度或自动发送规则。
   - RAG、LLM Wiki、Knowledge Card、Hybrid Retrieval、Graph、Rerank 或知识缺口流程。
   - OpenClaw、订单查询、风控、转人工、数据库表或队列行为。
   - 管理后台主要操作步骤和项目已知限制。
3. 文档必须区分“当前已经实现”“仅 Mock 验证”“仍待真实账号或生产验证”，禁止把占位 Adapter 描述成正式平台接口。
4. 每个阶段完成检查时，除类型检查和最小测试外，还要确认 `docs/project-flow.html` 的内容与代码一致、中文可读、UTF-8 无 BOM。
5. 只修改样式、错别字或与整体流程无关的局部实现时，可以不更新流程内容；如果无法确定是否影响流程，默认需要更新。
6. `docs/project-files.js` 保存 `docs/project-flow.html` 中的“文件职责索引”。新增、删除、移动、重命名核心文件，或改变文件主要职责时，必须同步更新该索引中的路径、负责内容和联动关注项。
