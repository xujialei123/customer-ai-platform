怎么用这份文档

这个项目大部分代码是按「平台接入 → 统一消息 → RAG/订单 → 草稿 → 回填发送」搭的。 不要从某个平台页面或某个选择器开始读，否则会把兼容旧代码和当前主链路搅在一起。 建议每天只做 1～2 步，打开文件时只问三件事：它从哪来、干什么、把结果交给谁。

主链路：扩展/Webhook → API → Worker → RAG → 草稿 → 插件发送 必读优先：带「必读」标签的文件 可跳过：Playwright 旧 watcher、旧 knowledge 兼容层

Chrome 插件：读 DOM、切会话、回填、点发送。

项目流程总览 部署手册 AGENTS.md 开发约束 本地引导页 /guide
1

规则与边界（先读，再动代码）

先弄清「允许做什么 / 禁止做什么」，避免梳理时被误导成可以自动乱发消息。

目标：理解 UTF-8、风控、默认不自动发送、Adapter 必须统一接入。

2

启动与基础设施

搞清楚 pnpm dev 到底起了哪些进程，数据存在哪。

目标：能自己画出「Docker / API 3001 / RAG 8787 / Mock 3100 / OpenClaw」关系图。

3

主 API 入口

从 Fastify 启动点往下看：路由注册、环境变量、数据库、队列。

目标：打开任意路由文件时，知道它挂在 main.ts 的哪一层。

4

消息主链路：Adapter → 会话 → 入库

所有平台消息最终都变成 UnifiedMessage，再进入同一套会话隔离。

目标：能手绘「抖音插件 inbound」与「企微 webhook」如何汇合到 MessageService。

5

ReplyWorker：回复大脑

队列消费者：订单优先，否则 RAG，再 OpenClaw，再风控，最后写草稿。

目标：看懂一次「你好」从队列到草稿的分支判断。

6

RAG 知识库（8787）

知识上传、Wiki/卡片编译、Hybrid 检索、回答生成。这是「AI 不能瞎编」的根基。

目标：分清「编译入库」和「在线检索回答」两条线。

7

Chrome 扩展 RPA（当前默认）

美团经营宝 / 抖音来客：在已登录浏览器里读 DOM，经 WebSocket 连本地 API。

目标：能说明「谁扫描消息、谁推草稿、谁点发送」，以及白名单如何生效。

8

企微、订单、风控配置

官方接口通路与「不能胡编订单状态」的安全边界。

目标：区分官方 Webhook 与 RPA；订单查询失败必须转人工话术。

9

交付、文档与样例

便携包脚本、流程文档、测试知识库。梳理完主链路后再看。

目标：知道给客户交付时改哪些文件，以及文档如何保持同步。
