二、角色分工（文件级）

文件角色在链路里干什么不干什么
manifest.json 清单 声明权限、能注入哪些网站、谁当后台脚本 不含业务逻辑
platform-profiles.js 配置 按 URL 判断美团还是抖音；给出默认 CSS 选择器 不连网、不点页面
background.js 中转 连本机中台；记「客户在哪个标签页」；把草稿送到对的页 不读聊天气泡、不点发送
content.js 核心 扫未读、切会话、上报消息、回填、点发送、串行锁 不调大模型
douyin-main-click.js 辅助 抖音点左侧客户时，在网页自己环境再点一次（防点不动） 不管美团、不管 AI
popup.html/js/css 遥控 开关监听/自动切会话/自动发送；改选择器；看连接状态 不跑主扫描循环

逻辑判断 · connect()
  • 如果 弹窗关了「启用页面监听」(enabled === false)
    → 状态标 disabled，不连中台
  • 如果 已经在连着 / 已连上
    → 直接 return，不建第二条 WebSocket
  • 如果 连上成功
    → 告诉中台「能不能自动发送」；清空旧会话登记；把对照表里还记得的客户重新 hello；给平台页补注入脚本
  • 如果 连接断开
    → 3 秒后再 connect；另有约 30 秒闹钟兜底（Chrome 后台睡着也能醒）

打开美团/抖音页时

webNavigation.onCompleted：只对经营宝、来客域名注入 platform-profiles.js + content.js；抖音额外注入 MAIN 世界的 douyin-main-click.js。

四、主链路 ② 扫消息：content 看见客户说话

入口：scanMessages()（MutationObserver 防抖 400ms + 每 1.5s 定时扫）。

4.1 每一轮扫描先过这些门

逻辑判断 · scanMessages 开头
  • 如果 扩展刚被重载，上下文已死
    → 停观察器，整轮结束
  • 如果 URL 不是美团/抖音/Mock
    → 跳过这个 document
  • 如果 enabled === false
    → 不上报消息（仍可能报诊断）
  • 如果 抖音停在「历史咨询」Tab
    → 清会话、不自动处理
  • 如果 客户不在白名单
    → 忽略并放锁
  • 如果 抖音聊天区还在 loading / 会话已关闭
    → 不读气泡（防串会话），但可继续调度切未读

4.2 过门之后上报什么

发给 background 的 type含义background 再干什么
session当前聊的是谁写入对照表；转中台 hello
inbound客户气泡正文原样转中台入库、排队 AI
outbound商家已发出的气泡转中台，用来确认「真的发出去了」
diagnostics选择器命中数、开关转中台给 /guide 看
逻辑判断 · 要不要把这条气泡当「新消息」入库
  • 如果 本会话第一次扫到（初扫）
    → 历史只记 seen，一般不入队（防打开页面把整天记录全回一遍）；有未读角标时末尾 N 条仍可入队
  • 如果 是抖音系统提示（新用户进线等）
    → 只记 seen，不当客户提问
  • 如果 同正文指纹已提交过
    → 跳过（切回话回扫不重复回）
  • 如果 抖音流式残文（「你」→「你好」）
    → 短的跳过，变长的允许更新

五、主链路 ③ 中台出草稿（插件外，但链路要接上）

插件把 inbound 送出后，大脑在本机中台：

入库去重 → ReplyWorker → RAG 查知识 → LLM 写回复
  → 风控（高风险转人工）→ 生成 ReplyDraft
  → extension-gateway 经 WebSocket 推 type=draft

推回来时带上 session（哪个客户）和 payload（回复正文、能否自动发、风险等级）。

六、主链路 ④ 回填 / 发送：草稿回到网页

6.1 background 怎么决定送给谁

逻辑判断 · 收到中台 draft
  • 如果 resolveDraftTarget 在对照表找到了 tab/frame
    → 精确 sendMessage(applyDraft) 到那个小页面
  • 否则 且是抖音
    → 向所有来客标签页广播 applyDraft（名字对不齐时的兜底），由 content 自己再切对客户
  • 否则 且是美团
    → 找不到目标就丢了（美团 id 通常更稳，一般靠精确路由）

6.2 content.applyDraft 怎么决定填不填、发不发

逻辑判断 · applyDraft
  • 如果 抖音在「历史咨询」
    → 不回填，回报 denyReason=history_tab
  • 如果 强制切到草稿客户失败
    → 回报 switch_failed，放锁，稍后重扫
  • 如果 找不到输入框 / 聊天未就绪
    → 回报 chat_not_ready 等，不硬点发送
  • 如果 页面上「本轮提问之后」已有同文商家气泡
    → 当作已发出，不再点发送（防切回话重发）
  • 如果 弹窗开了自动发送 且 中台允许 且 风险 low 且 有发送按钮选择器
    → 填框后点发送
  • 否则
    → 只填输入框（fill-only），等人点发送或等下次条件满足
回填前会清洗「微信/QQ」和「美团还是抖音买的吗」等话术，避免平台拒发或答非所问。

七、支线 A：自动切未读（多客户排队）

开关：弹窗「自动处理未读会话」→ autoSwitchConversations。核心在 content 的 scheduleUnreadConversation。

串行锁 conversationTask 三个阶段

phase意思能不能去切下一个未读
switching正在点左侧客户不能
waiting-draft已进会话，等 AI 草稿（最长约 120 秒）不能（超时才放）
applying-draft草稿正在回填/发送不能
null（空闲）没事干可以 schedule 下一个
逻辑判断 · 要不要切到某个未读
  • 如果 没开自动切会话
    → 整段调度直接 return
  • 如果 输入框里还有没发出的字（人在编辑）
    → shouldDeferSwitch，暂不切
  • 如果 抖音在历史咨询
    → 不切
  • 如果 当前中间已经是这个未读客户
    → shouldSwitchToUnread 返回 null，不用切
  • 如果 抖音聊天未就绪 / 停在系统会话 / 角标客户≠中间客户
    → 返回目标卡片，去点左侧
  • 如果 美团：左侧未读 id ≠ 当前会话 id
    → 切过去

抖音点击：content 先点一遍，再发 douyinMainClick → background 注入 MAIN 脚本再点（双保险）。

八、支线 B：找回草稿（不重跑 AI）

场景：你从 A 切到 B 又切回 A，A 还有未回完的话；或同页仍显示「尾巴未回复」。

逻辑判断 · maybeRecoverDraftsOnRefocus（重要！）
  • 如果 调度器已经在切「别的客户」
    → 不要抢锁（否则会出现「刚要点下一个未读，又被当前页钉死」）
  • 如果 左侧角标或「商家最后一句之后的客户句」> 0
    → 请求找回草稿；同会话最多约 8 秒一次
  • 如果 抖音找不到任何商家气泡
    → 待回复条数按 0 算（避免整页历史锁死自动切）

九、manifest.json 清单

没有 if/else 业务，但决定「能不能跑」：

  • host_permissions：能碰 3001、美团、抖音、Mock
  • content_scripts：匹配到的页自动注入 profiles → content
  • background.service_worker：后台中转是 background.js
  • action.default_popup：点图标打开 popup
改域名权限或版本后，必须到 chrome://extensions 点「重新加载」。

十、platform-profiles.js 配置

逻辑判断 · detectPlatformFromUrl
  • 如果 URL 含 life.douyin.com → douyin
  • 如果 含 g.dianping.com / ecom.meituan.com → meituan
  • 如果 是本机 :3100 Mock → 看 ?platform= 参数，默认抖音
  • 否则 → null（content 不处理）

mergePlatformSettings：全局开关（监听/自动切/自动发）+ 该平台选择器 + 代码默认值，合成 content 用的 settings。

十一、background.js 中转

文件头有「按时间读」导读。这里按链路归纳判断：

时机关键判断结果
连中台enabled？已有连接？不连 / 复用 / 新建 WS
收到 draft对照表有没有这个客户？精确投递 / 抖音广播 / 丢掉
收到 session同一小页面是否换人了？删旧登记 + 登新 + hello
settingsChanged（弹窗改了开关）断线重连 + 重新注入页面
douyinMainClick有没有 tabId？MAIN 世界执行点击脚本

对照表两张：

  • sessionTargets：客户名/id → 哪个标签页（送草稿用）
  • frameSessions：这个小页面现在聊谁（换人时清旧的）

十二、content.js 核心

最大、分支最多。按「主循环里发生的事」记：

12.1 每 1.5 秒（或 DOM 一变）发生什么

scanMessages
  ├─ 诊断 diagnostics → background
  ├─ scheduleUnreadConversation   // 自动切未读？看串行锁
  ├─ readSession                  // 现在是谁
  ├─ 白名单 / 历史 Tab / chatReady 守卫
  ├─ maybeRecoverDraftsOnRefocus  // 要不要找回草稿
  └─ 扫 inbound / outbound → 上报

12.2 串行锁（防多客户打架）

同时只服务一个客户。AI 慢时绝不能因为左边又亮了未读就跳走——否则当前客户的回复会填错框。

12.3 美团 vs 抖音差异（逻辑层）

点美团抖音
会话 iddata-user-id 等较稳多为昵称；内部 id 常对不上
切会话点击隔离世界点击即可再加 MAIN 世界补点
历史咨询—整段 RPA 暂停
气泡 id常有 data-messageid指纹：会话+方向+序号+正文
连发多句按角标末尾 N 条冷切入取末尾 N；服务中可全量扫

十三、douyin-main-click.js 辅助

逻辑判断
  • 如果 在历史咨询 Tab → 不点，返回 history_tab
  • 如果 找不到卡片 → no_cards
  • 如果 有 nameHint → 按昵称模糊找卡再点
  • 否则 → 优先点「客户未读」（跳过预警通知），再退回任意未读

十四、popup.html / popup.js / popup.css 遥控

控件关掉会怎样打开会怎样
启用页面监听background 不连中台 / content 不处理业务正常扫和上报
自动处理未读会话不自动点左侧，只处理你当前停在的客户串行切未读
允许自动点击发送只填输入框low 风险且中台允许时可点发送

保存后发 settingsChanged → background 重连并 reinject。AI 自动识别：采脱敏 DOM → 打本机 API → 校验后再填选择器。
