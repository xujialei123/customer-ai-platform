一、总览

扩展跑在你已登录的美团经营宝 / 抖音来客页里。它不登录平台，只读 DOM、上报消息、接收草稿并回填或点击发送。大脑（RAG/LLM）都在本地 API；扩展和 API 用本机 WebSocket 说话。

二、扩展内文件分工

content.js

页面脚本（核心）

注入聊天页：扫未读、切会话、读气泡、上报 inbound/outbound、收到草稿后回填/点击。有串行锁 conversationTask。

background.js

后台 Service Worker

连 ws://127.0.0.1:3001/rpa/extension/ws；把 content 消息转发网关；把 draft 路由回正确标签/frame。

popup + profiles

开关与选择器

启用监听、自动处理未读、自动发送；美团/抖音选择器与白名单。自动发送还须服务端 RPA_AUTO_SEND_ENABLED。

三、一条消息从进到出（主链路）

1. content 定时/DOM 变化 → scanMessages()
2. 发现未读 → scheduleUnreadConversation() 切到该客户（串行）
3. 读客户气泡 → 生成稳定 messageId → 发 type=inbound
4. background → Gateway：先登记 expectedMessageId，再 POST /rpa/inbound
5. ReplyWorker：订单路由 / RAG / LLM → 写 ReplyDraft
6. Gateway 约 1.5s 轮询：matched 或孤儿救援 → 推 type=draft
7. background 按会话找到 tab → content.applyDraft()
8. 强制切回草稿对应会话 → 回填输入框
9. 若允许自动发送 → 点发送按钮 → draft_send_result
10. 页面再扫到商家气泡 → type=outbound → 草稿标 sent
安全默认：没开弹窗「自动发送」或没开环境变量 RPA_AUTO_SEND_ENABLED 时，只回填不点击，需人工确认再发。

四、会话串行锁 conversationTask

同时只能服务一个「当前任务」，防止 A 等回复时跳到 B 把答案填进错会话。

switching

正在切换

刚点了左侧会话，还在等聊天区落地。卡住过久会清锁重试。

waiting-draft

等待 AI 草稿

已在目标会话（或切入后升级）。其它未读排队，约 120 秒超时才放锁。

applying-draft

草稿投递中

草稿到了，强制切回该客户再回填/发送。期间禁止未读抢切。

放锁时机

  • 回填/发送结束 → releaseConversationTaskAfterDraft → 过一会再扫下一个未读
  • 切失败 / 等草稿超时 → clearConversationTask

五、expected 是什么（网关侧）

一句话

expectedMessageIds = 本插件连接当前在等「哪几条客户消息」的 AI 回复。inbound 上报时先写入，再等草稿 messageId 对上再推。

为什么需要

LLM 很慢时库里可能堆多条 pending。没有 expected，旧答会被推到当前会话造成乱回/重发。

选稿优先级

unrepliedDrafts（pending 且尚未被判定已回复）
    ├─ matchedDrafts：draft.messageId ∈ expected     ← 优先
    └─ recentOrphanRescue：近期孤儿（expected 对不上时的救援）
candidates = matched 按时间 FIFO；自动发送每轮最早 1 条。expected 空时孤儿/pending 同样从早到晚兜底。
自动发送开启时每轮最多推 1 条

六、扫描与入库（content.js）

步骤做什么注意点
读 session美团埋点 userId / 抖音昵称+URL系统会话、历史咨询 Tab 不处理
切回话待回复检测尾巴未回复；有 pending 草稿则 request_drafts 直接推送发送不重新跑 LLM；若调度器已在切其他未读，禁止用当前页找回抢锁（0.3.29）
初扫入队美团/抖音均按未读角标放行末尾 N 条（不是只最后 1 条）同客户连发中间句不能只记 seen
稳定 messageId优先 DOM 显式 id；否则「会话+方向+出现序号+正文」不用 DOM index，避免切回话变新 id 重复回复
同正文去重submittedInboundContents切回话回扫同一句不二次入库
初扫只记 seen，不入队避免一打开页面就把历史全回一遍
outbound商家气泡上报用于确认真正发出 → 草稿标 sent

何时会点发送（四者都要）

  • 弹窗勾了自动发送
  • 服务端 RPA_AUTO_SEND_ENABLED=true
  • 草稿 riskLevel === 'low' 且 allowAutoSend
  • 有发送按钮选择器，且本轮提问后页面还没有同文商家气泡
踩过的坑：超时兜底话术经常同一句。若「整页任意历史有同文就跳过」，会误判 already_on_page 导致新问题完全不发。现已改为只看「当前客户这句提问之后」。

八、网关侧关键概念

名字含义
expectedMessageIds当前连接等待回复的客户消息 id 集合
matchedDraftsmessageId 命中 expected 的草稿
recentOrphanRescueexpected 对不上时的近期孤儿救援名单
draftPushState每条草稿推过没有、点过没有，防重复推/重复点
denyReason为什么不能自动发送（扩展开关/服务端开关/风险/冷却等）
fill_only只推到输入框，不授权点击

九、扩展 ↔ 网关消息类型

扩展 → 网关
  • hello / session：登记会话
  • inbound / outbound：客户/商家消息
  • client_settings：同步自动发送开关
  • draft_send_result：是否点到发送
  • diagnostics：选择器诊断（无正文）
  • heartbeat / reset_sessions
网关 → 扩展
  • connected：握手 + 白名单状态
  • draft：推送待回填/发送的草稿
  • inbound_ack / outbound_ack
  • heartbeat_ack / error

十、终端日志 ↔ 含义

日志/现象大致含义先查什么
pending=N expected=0有草稿但对不上 expected孤儿救援是否推动；扩展是否重连丢名单
LLM 超时，本地兜底等满超时后用本地话术Agnes 延迟；OPENCLAW_TIMEOUT_MS
推送插件 自动发送=是网关已授权点击接着应有「已点发送」或点击失败
already_on_page认为本轮后已有同文，跳过点击是否误判历史同文；扩展是否 ≥0.3.23
fill_only只回填不点弹窗开关 / RPA_AUTO_SEND_ENABLED / 风险等级
切回话又重发旧稿再推或 messageId 抖动串行锁、draftPushState、稳定 id
