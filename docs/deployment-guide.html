一、上线前检查

必需软件

Node.js 20+、pnpm 9.15、Docker Desktop、Google Chrome。源码部署还需要 PostgreSQL/pgvector 与 Redis，默认可由 Docker Compose 提供。

端口

3001 API、8787 RAG、5433 PostgreSQL、6379 Redis。本机 OpenClaw 18789 仅在 LLM_PROVIDER=openclaw 时需要。

安全默认值

AUTO_REPLY_ENABLED=false、RPA_AUTO_SEND_ENABLED=false、RPA_DRY_RUN=true。首次上线不得跳过灰度。

模型配置：默认直连 Agnes 或自定义 OpenAI 兼容接口。优先在 http://127.0.0.1:3001/guide 配置 LLM 与 Embedding（热生效）。便携包不再捆绑 OpenClaw，仅携带 runtime\node-win-x64 用于启动 API/RAG。
密钥边界：模型 Key、平台 Token、Cookie、账号密码只放部署机环境变量、配置页或密钥系统，不写入仓库和对外交付包。完整 API Key 不会回传给前端。

二、源码开发部署

  1. 复制 .env.example 为 .env，或启动后到 /guide 填写 LLM / Embedding。
  2. 启动依赖并迁移数据库。
  3. 构建全部工作区，再启动服务。
chcp 65001
$OutputEncoding = [System.Text.Encoding]::UTF8
Copy-Item .env.example .env
pnpm install
docker compose up -d
pnpm prisma:migrate
pnpm build:all
pnpm dev
最小健康检查：http://127.0.0.1:3001/health、http://127.0.0.1:3001/health/llm、http://127.0.0.1:3001/health/embedding、http://127.0.0.1:8787/health 均应正常；配置引导页 http://127.0.0.1:3001/guide；知识库管理页 http://127.0.0.1:8787/kb-admin。
注意：不要从便携包测试环境带走 CUSTOMER_AI_ROOT 环境变量再跑源码 pnpm dev，否则会误读便携包 .env。开发编排器会强制把 CUSTOMER_AI_ROOT 指回仓库根目录。

三、Windows 便携交付

pnpm build:all
pnpm package:windows
  1. 从 release\Customer-AI-Portable-YYYYMMDD-HHmmss 取完整目录，不拆散内部结构。
  2. 对外交付建议加 -SafeSendDefaults，并避免把本机模型 Key 打进包。
  3. 客户机安装 Docker Desktop 与 Chrome，双击 Start-Customer-AI.bat。
  4. 打开引导页 http://127.0.0.1:3001/guide：配置 Agnes 或自定义 LLM、Embedding、白名单；再于 chrome://extensions 加载 extensions\customer-ai-rpa。
  5. 停止使用 Stop-Customer-AI.bat，诊断使用 Doctor-Customer-AI.bat。
便携包是 Windows 单机交付方式，不等同于高可用生产集群。数据库卷、知识库和浏览器 Profile 必须纳入客户机备份。

四、生产部署基线

组件生产要求验收
API / ReplyWorker建议拆分进程；配置进程守护、结构化日志和失败告警。进程重启后消息不重复生成草稿。
PostgreSQL启用 pgvector；独立账号、最小权限、每日备份和恢复演练。迁移完成，业务表与 RAG 表均可读写。
Redis启用持久化与访问控制；禁止暴露公网。BullMQ waiting/active/failed 可观测。
RAG / 模型使用真实 Embedding 与 LLM（Agnes 或自定义兼容接口）；设置超时和调用配额；维度与 EMBEDDING_DIM 一致。门店问题可召回证据，无证据固定转人工；/guide 可查看与热切换配置。
入口安全外部 Webhook 使用 HTTPS 反向代理；校验签名、限制来源和请求大小。伪造回调被拒绝，健康接口不泄密。
当前 Chrome 扩展 RPA 依赖客服电脑的已登录页面。若需要无人值守、多机容灾或集中调度，需另行建设持久化租约、心跳选主、发送审计和人工接管工作台后再宣称生产高可用。

五、RPA 灰度上线

  1. 保持服务端和插件自动发送均关闭，只验证页面识别、会话 ID、inbound 入库和草稿回填。
  2. 在 http://127.0.0.1:3001/guide 为美团、抖音分别配置测试客户白名单。空白名单表示全量开放，灰度期禁止留空。
  3. 用退款、投诉、赔偿、食品安全、无知识库答案等问题验证必须转人工。
  4. 服务端设置 RPA_AUTO_SEND_ENABLED=true 并重启，再仅对测试客户打开插件自动发送。
  5. 核对草稿状态：pending → dispatching → sent。点击按钮只进入 dispatching；页面观察到真实 outbound 才算 sent。
  6. 至少完成跨门店、同名客户、页面切换、断网重连、刷新、连续消息和人工抢答测试，再逐步扩大白名单。
立即停发条件：串会话、重复发送、高风险自动发送、知识库外编造、页面选择器漂移或 outbound 无法确认。关闭插件开关与 RPA_AUTO_SEND_ENABLED 后再排查。

六、验收、备份与回滚

便携包 RPA 专项验收清单见包内 portable-test-checklist.html（打包时随 packaging/windows-portable 一并复制）。

发布验收

pnpm --filter @customer-ai/api test
pnpm build:all
  • 检查 docs/project-flow.html 与本手册能直接打开、中文可读、UTF-8 无 BOM。
  • 重复投递同一 messageId，只入库一次且只产生一份队列任务/草稿。
  • 白名单外客户不入库、不排队、不回填、不发送。
  • RAG 失败、LLM 失败和高风险问题均保留人工草稿，不伪装成功。
  • 在 /guide 切换 Agnes / 自定义后，新消息走新模型；Embedding 变更后注意维度并重编译知识库。

备份

备份 PostgreSQL、Redis 持久化数据（如启用）、知识库上传目录、config/model.local.json（含密钥，需加密）、运行时配置与 Chrome Profile。密钥备份应使用独立加密介质。

回滚

  1. 先关闭两个自动发送开关，保留人工接管。
  2. 停止应用进程，恢复上一版本应用目录和匹配的环境配置。
  3. 数据库迁移仅按已评审的回滚 SQL 执行；不使用破坏性 reset。
  4. 重新运行健康检查和只读链路测试，最后再恢复白名单灰度。
