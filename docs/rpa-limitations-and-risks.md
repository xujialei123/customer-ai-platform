# RPA 方案局限、难点与当前问题

## 1. 文档目的

本文记录抖音、美团浏览器 RPA 方案的能力边界、实现难点、已遇问题和生产上线条件，供项目汇报、技术评审、测试和后续维护使用。

当前结论：美团经营宝已经跑通“当前活动会话的新消息采集、后端生成草稿、浏览器输入框回填”链路，但尚未完成左侧多客户未读会话调度，不能视为无人值守的生产级自动客服。

## 2. 当前方案

```text
普通 Chrome 中的经营宝登录会话
  -> Chrome 扩展读取 DOM / Shadow DOM
  -> 本地 WebSocket Adapter
  -> UnifiedMessage 去重入库
  -> 订单系统查询或 RAG 知识库检索
  -> OpenClaw 生成回复草稿
  -> SafetyService 风控
  -> WebSocket 返回对应客户会话
  -> 扩展回填经营宝输入框
  -> 人工确认发送
```

使用普通 Chrome 是为了复用人工正常登录的会话，避免由 Playwright 打开登录页时触发平台风控。扩展不负责登录，不读取账号、密码或 Cookie。

## 3. 当前已实现能力

- 识别经营宝普通 DOM 和封闭 Shadow DOM。
- 只采集客户侧 `left-message`，排除商家侧 `right-message`。
- 使用平台 `messageId` 和后端数据库进行两层去重。
- 从页面埋点读取真实 `shopId`、`userId` 和客户名称。
- 首次进入会话只建立历史消息基线，不重新处理屏幕中的历史消息。
- 草稿回填前再次校验门店和客户会话，降低串话风险。
- 订单问题可进入公司订单系统 Adapter，普通问题进入 RAG 和 OpenClaw。
- 自动发送默认关闭，高风险问题必须转人工。
- 提供脱敏 DOM 快照、OpenClaw 候选选择器和浏览器现场验证流程。

## 4. 尚未完成的问题

### 4.1 左侧多客户新消息尚未调度

当前只处理正在打开的客户会话。左侧其他客户出现新消息时，聊天正文通常尚未加载到主聊天区域，因此扩展无法直接读取完整消息。

后续需要实现：

- 监听左侧会话列表的未读标记、最新消息和列表变化。
- 将未读客户加入待处理队列并去重。
- 依次打开会话，等待真实 `userId` 和消息 DOM 加载。
- 读取新消息后恢复或切换到下一个会话。
- 对队列设置超时、失败重试、最大重试次数和死信记录。

这是当前距离“多用户并发客服”最大的功能缺口。

### 4.2 自动切换会干扰人工客服

如果 RPA 和人工客服共用一个经营宝标签页，自动点击左侧客户会改变当前会话，可能导致人工正在编辑的内容丢失或发错客户。

建议使用独立 Chrome 窗口或专用标签页运行 RPA，并增加以下保护：

- 检测输入框是否有人工未发送内容，有内容时暂停切换。
- 检测鼠标、键盘和页面焦点活动，人工操作期间暂停队列。
- 切换前后校验 `shopId`、`userId` 和客户名称。
- 回复草稿必须绑定目标客户 ID，禁止只按当前页面发送。

### 4.3 自动发送尚未达到生产条件

目前默认只回填输入框，不自动点击发送。自动发送需要同时满足扩展开关、服务端开关和风控结果，但即使满足这些条件，也仍需完成多客户调度、审计和灰度验证后才能考虑开放。

退款、投诉、差评、赔偿、食品安全、法律纠纷和情绪激烈等内容不得无人审核自动发送。

## 5. 实现难点

### 5.1 页面结构不是稳定接口

经营宝 DOM、class、节点层级和渲染方式可能随平台发布变化。RPA 选择器不是官方接口契约，今天有效不代表长期有效。

当前提供 AI 自动识别功能，但 AI 只能生成候选选择器。候选结果必须在真实页面验证以下条件后才能保存：

- 客户消息命中数量大于 0。
- 不得命中商家侧消息。
- 每条消息都能找到文本节点。
- 回复输入框只能命中 1 个。
- 发送按钮只能命中 1 个。
- 当前客户会话根节点只能命中 1 个。

### 5.2 Shadow DOM 增加采集复杂度

经营宝聊天内容位于封闭 Shadow DOM 中，普通 `document.querySelector` 和常规 iframe 注入无法读取。当前通过 Chrome 扩展的 Shadow Root 能力递归扫描。

递归扫描的弊端：

- 页面节点较多时可能增加 CPU 消耗。
- 平台改为跨域隔离、受限 frame 或其他渲染容器后，现有方式仍可能失效。
- 扫描频率过高会影响经营宝页面性能，过低则会增加回复延迟。

### 5.3 虚拟列表和延迟渲染

左侧会话列表和消息列表可能使用虚拟滚动，只在 DOM 中保留可见项目。未滚动到的客户或历史消息可能根本不存在于当前 DOM，因此不能通过一次全页面扫描获得全部会话。

### 5.4 登录态和平台风控

此前使用 Playwright persistent context 登录经营宝时，密码登录出现“请求异常”，疑似触发平台自动化风控。因此改为普通 Chrome 登录加扩展采集。

该方案仍不能绕过验证码、账号限制或平台风控，也不应尝试伪装自动化特征。登录失效、异地验证和平台安全策略变化仍需人工处理。

### 5.5 OpenClaw 输出并非总是严格结构化

实测 OpenClaw 可能未返回符合约定的选择器 JSON，或者请求超时。当前做法是：

- DOM 分析最多等待 8 秒。
- 使用 Zod 校验模型 JSON。
- 模型失败时，只允许根据快照中实际存在的稳定 class 生成本地候选。
- 无论 AI 还是本地候选，都必须通过浏览器现场验证。
- 验证失败时保留上一版配置，禁止覆盖。

### 5.6 WebSocket 和浏览器生命周期

Chrome Manifest V3 Service Worker 会休眠或重启，本地 API、浏览器标签页和扩展也可能分别重启。需要依靠心跳、断线重连和会话重新注册保持状态。

电脑重启、`pnpm dev` 重启、扩展重新加载后，都应重新验证连接和当前会话，不能假设旧内存状态仍然有效。

## 6. 已遇到的具体问题

### 6.1 Playwright 登录触发请求异常

表现：服务启动的专用浏览器中输入经营宝账号密码后提示“请求异常”。

处理：停止把 Playwright 作为默认登录入口，改为普通 Chrome 登录和扩展 Adapter。Playwright 模式只保留兼容代码，不随默认开发启动流程运行。

### 6.2 普通 iframe 注入无法获得聊天 DOM

表现：扩展显示已连接，但消息、输入框和发送按钮命中数始终为 0。

原因：聊天内容不是普通可访问 iframe，而是封闭 Shadow DOM。

处理：增加 Shadow DOM 递归扫描后，真实页面命中客户消息 3 条、输入框 1 个、发送按钮 1 个，并读到真实 `shopId` 和 `userId`。

### 6.3 占位会话旧草稿误回填

表现：输入框自动出现“这个问题我帮您转人工确认一下”，但真实客户会话下查不到对应新草稿。

原因：外层页面曾以页面标题注册为占位会话，旧草稿被路由到同一标签页。

处理：

- 只注册具有真实 `shopId/userId` 的会话。
- WebSocket 重连时清空旧会话注册。
- 回填前再次核对当前页面的门店 ID 和客户 ID。
- 外层壳页面不允许作为客户会话接收草稿。

该问题说明“标签页正确”不等于“客户会话正确”，后续所有发送动作必须坚持客户 ID 二次校验。

### 6.4 AI DOM 分析返回无效 JSON

表现：OpenClaw DOM 分析接口等待后返回无效结果。

处理：增加 8 秒超时、严格 Schema 校验、本地结构化候选和真实页面验证。模型失败不会覆盖当前配置。

## 7. 安全与隐私限制

- 不在代码、扩展或日志中保存平台账号、密码和 Cookie。
- DOM AI 快照不包含聊天正文、手机号、客户 ID、门店 ID、图片地址、元素 ID 或可疑数据属性值。
- `lx-mv` 只上传“属性存在”，不上传属性原值。
- 回复草稿必须按 `platform + shopId + conversationId` 隔离。
- 自动发送默认关闭，并要求扩展端和服务端同时开启。
- 高风险问题不得自动发送。
- 不能使用 RPA 绕过验证码、风控、权限或平台规则。

## 8. 生产上线风险

- 平台页面升级可能导致选择器失效或采集停止。
- 多客户自动切换可能与人工客服争抢页面控制权。
- 虚拟列表可能造成未读会话漏检。
- 网络、OpenClaw、RAG、Redis、PostgreSQL或订单系统任一故障都会影响链路。
- 服务重启可能造成内存队列丢失，关键调度状态应持久化。
- 如果缺少发送审计和幂等控制，重试可能导致重复回复。
- RPA 属于页面自动化方案，稳定性和合规性低于平台官方 API。

## 9. 建议实施顺序

1. 使用本地多会话 Mock 验证左侧未读监听、串行切换、消息采集和回复回填。
2. 在真实账号中确认会话列表、未读角标和虚拟滚动结构，并调整 Adapter 选择器。
3. 将当前浏览器内存调度升级为持久化队列和可恢复状态机。
4. 增加独立 RPA 标签页模式、人工操作暂停和原会话恢复机制。
5. 增加积压数量、失败次数、断线和选择器失效监控。
6. 在真实账号中进行小范围、只回填、不自动发送的灰度测试。
7. 完成重复发送防护和完整审计后，再评估低风险场景自动发送。
8. 持续确认平台是否提供适用于到店团购客服的正式接口；如有官方能力，应优先迁移到官方 API。

## 10. 对外汇报口径

当前 RPA 已通过 Chrome 插件 + 本地 WebSocket Adapter 完成多会话 Mock 的未读发现、串行切换、消息采集和回复回填，可用于无真实门店账号时的完整链路演示。当前调度队列仍在浏览器内存中，真实平台虚拟列表、人工操作协调、进程重启恢复和生产监控尚未完成，因此不能描述为已达到生产级无人值守客服。
